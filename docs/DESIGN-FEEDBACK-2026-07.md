## 一、先说结论：桌面零增量，价值全在手机

这套 DS 明确说是从 `public/reader.html` 反向工程出来的——所以桌面侧的几何、色板、字体**逐值一致**：58px 顶栏、312px 侧栏、720px 阅读列、3px 纸圆角（全站唯一不圆的那个，因为它是纸不是卡）、34px 行距信纸线、下沿 42% 的高亮 wash、爪印滤镜……**我一条都不用改**，因为线上本来就长这样。

真正有价值的是**你新做的那部分**：手机重设计（另一套 IA——底部三 tab + 沉浸满屏纸 + 悬浮 dock + 蜡书脊进度 + 三种纸）。那是现状里没有的东西。

**所以：如果目标是改进现状，反推现状再交回来收益有限；直接提新方案（像手机那套）价值高得多。** 反推的那部分留作文档很好（`readme.md` 把散落在 550 行 `reader.html` 里的规则提炼成了可查的体系，这个有独立价值），但不必当成交付物的主体。

## 二、DS 内部有两处自相矛盾（施工时真不知道听谁的）

这两条不是挑刺，是**卡住施工**的问题：

### 1. 纸到底翻不翻？

`readme.md` 白纸黑字写信纸是物理对象、`never fully inverts`——这是第 4 条公理（冰的留白）和整个「纸与界分层」哲学的基础。

但 **DS 自己的 `tokens/colors.css` 暗色段把 `--vk-paper` 翻成了冷灰 `#2C2E40`**。文档和 token 打架。（线上现状也是翻的。）

我这次**按现状保持不动**，没敢依 readme 去改。请裁决：改 readme，还是改 colors.css？

### 2. 底部第二个 tab 是信箱还是统计？

- `readme.md` 说三 tab 是「书架 / **信箱** / 我」
- `ui_kits/mitlesen-mobile/mobile-app.jsx` 实现的是「书架 / **统计** / 我」
- 而 `.letters` / `.sealed` 那套火漆信件 CSS **写了，但没有任何地方引用**——信箱有壳无芯

两条路工作量差一个量级：走统计只是本地数据；**走信箱要跨服务接 Nachklang 的 Vorklang 慢递系统**。手机批要等这条裁决才好动。

### 3.（附带）「换纸 ≠ 暗色模式」这个概念很好，但要在 token 层钉死

三张实体纸 warm / snow / night，night 纸的 `#23212C` 是**纸本身的颜色**，不是 UI 反色——这个区分我很喜欢，它跟「纸是物件」的哲学是自洽的。

但现在 night 纸和 `[data-theme=dark]` 在实现里混着走。建议在 token 层彻底分开：纸走 `--paper-*` 自己一套（由用户选纸驱动），主题只管 chrome（由系统/开关驱动）。分开之后上面第 1 条的矛盾也会自动消解。

## 三、有一个真实场景 kit 完全没覆盖

`ui_kits/mitlesen-reader` 的示例书只有 3 章。**线上真书 48 章**——目录在侧栏全部平铺，一屏塞不下，书架里的第二本书被挤到列表最底部（几乎发现不了）。

kit 里没有这个场景的设计，施工侧只能自己想（目录区独立 `max-height` + 内滚？折叠到「当前章 ±N」？）。这属于**超出设计授权范围**，我没敢自作主张。

同类提醒：请按真实数据量级设计，或在稿里标一句「N 很大时怎么退化」。（姊妹篇里 Klanggestalt 那边同一个病出现了三次：3 首歌 vs 156 首、3 个调 vs 25 个、4 簇 vs 17 簇。）

## 四、五公理是这套 DS 最值钱的东西

必须专门说一下：**五公理不是漂亮话，是可执行的判据。**

施工时我真的拿它当检查表用，然后靠第 5 条「一页一仪式」发现了一个**产品级缺失**：

> 共读的阅读器**根本没有仪式**。读完一章只弹一句 toast 就没了。
> 而后端 `GET /api/card-inbox`、`GET /api/cards/:id/image.svg`（服务端直接渲染好的 SVG）、`POST /api/cards/:id/dismiss`、`GET /api/card-collection` **全部现成躺着，没有任何前端在用**。

公理说「邮箱破火漆、阅读器铸仪式卡、记忆有时间尺、统计有山脉」——一比对就发现阅读器这一格是空的。**没有那条公理我不会去查这个。** 这已经列为桌面批的主菜（后端零改动，纯前端接线）。

其他几条同样能落到代码判断上：
- 「纸与界分层」→ 直接决定一个元素该用 `--vk-paper` 还是 `--surface`，不用猜
- 「衬线只在四个位置（标题/日期/落款/引文）」→ 见到第五处衬线就知道是错的
- 「蜡的物性：情绪不用色条、全部收敛到蜡点这一个形状」→ 挡住了「随手加个情绪色条」的冲动

**比「优雅、克制、有呼吸感」这类形容词有用一个数量级。** 请在别的项目里也这么写。

## 五、其他小结论

- **组件的 `.prompt.md` 三件套（`.jsx` + `.d.ts` + `.prompt.md`）很好用**——`.d.ts` 让 props 契约无歧义，`.prompt.md` 说清楚什么时候该用这个组件。施工时基本不用回头猜。
- **`SKILL.md` 做成 user-invocable skill 的形态是对的**，但请注意：它现在只活在一个 zip 里。建议把整套 DS **解压入库**（比如放 `design/` 目录）而不是以 zip 形式躺在仓库根部——zip 在 git 里是不可 diff 的二进制块，改一版就多存一份全量。
- 无彩色 emoji 这条守得很好，`🗑` / `📖🎉` 两个例外还专门在 Iconography 表里登记了——施工时看到就知道不用换。
